# Why Schemas

Without Schemas, our endpoint functions have to check everything themselves: check fields exists, check types, check ranges of values, and handle every possible malformed input. Schemas move all of those validation out of our lgic and into a declarative definition so that our endpints can assume that the data is already clean by the time they run.

# What a schema actually is

A schema is a BaseModel (from pydantic) subclass where each attribute is a typed contract:

```
class TodoBase(BaseModel):
  todo_name: str = Field(..., min_length=3, max_length=512, description='Name of the todo')
  todo_description: str = Field(..., description='Description of the todo')
  priority: Priority = Field(default=Priority.LOW, description='Priority of the todo')
```

# Why multiple Schema classes instead of one

Different operations need different shapes of data, even for the same resource like /todos or
/books and such.

```
Create(POST) - Clients sends name/description/priority; server assigns todo_id
Read(GET) - client receives everything, including todo_id
Update (PUT) - client should be able to send only the fields they're changing
```

Hence if you use one schema for all three, you get contradictions. For instance,
todo_id is required for the client to send when creating something, which is wrong - the server owns that.

# Pattern
```
class TodoBase(BaseModel):
  todo_name: str = Field(..., min_length=3, max_length=512, description='Name of the todo')
  todo_description: str = Field(..., description='Description of the todo')
  priority: Priority = Field(default=Priority.LOW, description='Priority of the todo')


class TodoCreate(TodoBase):
  pass

class Todo(TodoBase):
  todo_id: int = Field(..., description='Unique identifier of the todo')

class TodoUpdate(BaseModel):
  todo_name: Optional[str] = Field(None, min_length=3, max_length=512, description='Name of the todo')
  todo_description: Optional[str] = Field(None, description='Description of the todo')
  priority: Optional[Priority] = Field(None, description='Priority of the todo')

```

# from_attributes for Response_model True in Endpoint Schemas

```
class Config:
  from_attributes = True
```
This tells pydantic schema that it is allowed to build itself from an object's attribute:
obj.field, and not just from a dict (obj['field']). This is required on any schema that is used as a response_model.
```
@app.get('/people/{person_id}', response_model=PersonRead)
def get_person(person_id: int, db: Session = Depends(get_db)):
    person = db.get(Person, person_id)
    return person
```
When our endpoint has response_model=PersonRead, FastAPI does not just take whatever we return and send it out as a JSON.
- It returns person - a Person ORM object.
- FastAPI takes that return value and needs to convert it into a PersonRead instance, since that is what response_model promised.
- It does something like: PersonRead.model_validate(person) - This builds the response and constructs an actual PersonRead object out of whatever person is. If we did not have from_attributes = True in the schema. the validate tries to read data the default way by treating person like a dict looking for person['id'], etc. With from_attributes, it reads data the attribut way - person.id.

# Input vs Output Schema

```
class PersonCreate(BaseModel):     # <- INPUT schema
    name: str
    age: int | None = None
    # NO Config needed


class PersonRead(BaseModel):       # <- OUTPUT schema
    id: int
    name: str
    age: int | None

    class Config:
        from_attributes = True     # <- needed
```

- PersonCreate does not need class Config within its Schema because it is already built from the incoming JSON request body - the client sends in a JSON dictionary, not an ORM object.
- PersonRead does need it since the thing being converted into the required response_model is an ORM instance - accessed via .name, .age attributes not dict keys.


- By inheriting BaseModel, our class automatically gets Validation for types match, coerce compatible types, autogenerated schema/docs, builds OpenAPI schema shown in /docs.

- TodoBase holds the common contract. Todo inherits it and adds the server-assigned todo_id - that is our response model. TodoUpdate intentionally does not inherit TodoBase, because every field needs to flip from required in TodoBase to Optional in TodoUpdate, which is a different contract. The TodoUpdate definition is a pydantic model meant for partial updates. Every field is optional so a client can update just one field without resending the rest.

- Field(None, ...) - the default is None so the field isn't required in the request body at all and can be omitted entirely.
